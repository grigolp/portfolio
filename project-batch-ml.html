The Challenge

Time-series forecasting for retail sales requires predictions across multiple future time periods (1-18 months ahead), with each horizon requiring independent model training.

Business Constraints

  • Sequential training: 30 days per full model refresh (18 models × ~40 hours each)
  • Zero-inflated data: Many zero sales periods requiring specialized handling
  • Large datasets: ~3GB of training data with 825+ engineered features
  • Frequent retraining: Models need updates as new data arrives monthly
  • Cost sensitivity: Must balance compute speed vs infrastructure costs

Technical Requirements

  • ✓ Scalability: Train 18 independent models in parallel
  • ✓ Automation: End-to-end deployment without manual intervention
  • ✓ Reliability: Graceful error handling, comprehensive logging
  • ✓ Cost Efficiency: Optimize for AWS compute costs
  • ✓ Maintainability: Clean architecture, configuration-driven design
  • ✓ Reproducibility: Version-controlled configs and containerization

Key Design Decisions

1. AWS Batch Array Jobs

Rationale: Native parallel processing without manual orchestration

Implementation: 1 parent job spawns 18 child tasks, each training 1 horizon

Impact: 30 days → 12 hours (limited by longest-running task)

2. Spot Instance Strategy

Rationale: 60-70% cost savings for interruptible workloads

Strategy: Checkpoint model training every 10 iterations, resume on interruption

Impact: ~$750/cycle → ~$225/cycle (with <5% interruption rate)

3. Docker Containerization

Rationale: Reproducible environments across dev/production

Optimization: Base image mambaorg/micromamba:1.5 for 60% faster dependency resolution

Impact: 1.8GB final image with all ML dependencies, cached for instant deployment

4. Pydantic Configuration Validation

Rationale: Type-safe configs prevent runtime errors in production

Implementation: All YAML configs validated against Pydantic schemas

Impact: Zero config-related failures in 50+ production runs
