Key Design Decisions

1. Iterative Removal Algorithm (O(VÂ²) vs O(V!))

Rationale: Traditional BFS explores all possible vehicle removal orderings - 18 vehicles = 6.4 quadrillion states

Strategy: Key insight - if vehicles A and B can both exit in current state, removal order doesn't matter. Remove ALL movable vehicles per iteration.

Impact: Reduced from seconds to milliseconds, enabling real-time validation during level design

2. Pre-Calculated Path Lookup Table

Rationale: Runtime pathfinding for each vehicle check would be too slow

Strategy: Calculate all possible exit paths once during graph construction. Store in O(1) lookup table indexed by (node, orientation, movement_rule)

Impact: Instant path validation during solving phase, eliminates repeated pathfinding calculations

3. Multi-Layer Caching Strategy

Rationale: Same grid layouts used repeatedly (50-100 validations per level during design)

Strategy: MD5 hash-based graph caching (95% hit rate), occupancy caching (79% speedup), custom position hashing (27% speedup)

Impact: First validation: ~250ms cold start. Subsequent: <10ms with warm cache

# Novel O(VÂ²) iterative removal algorithm
def validate_puzzle(board, vehicles):
    solution_order = []

    while vehicles:
        # Find ALL vehicles that can exit right now
        movable = [v for v in vehicles if can_exit(v, board)]

        if not movable:
            return {"solvable": False, "reason": "Deadlock detected"}

        # Remove all movable vehicles (order doesn't matter)
        for vehicle in movable:
            remove_vehicle(vehicle, board)
            vehicles.remove(vehicle)
            solution_order.append(vehicle.id)

    return {"solvable": True, "solution": solution_order}

# Key insight: Order within iteration doesn't matter
# Only which iteration vehicles exit matters

Technical Challenges Solved

1. State Space Explosion (O(V!) â†’ O(VÂ²))
