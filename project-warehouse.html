The Challenge

Kasparium's warehouse model inverts traditional operations: racks move to workers at delivery stations rather than workers walking to racks. System must coordinate parallel rack movements for 100+ simultaneous orders without collisions or deadlocks.

Requirements & Constraints

  • Orchestrate parallel rack movements for 100+ simultaneous orders
  • Optimize delivery speed while respecting order priorities
  • Manage limited empty floor spaces efficiently
  • Handle dynamic station traffic with configurable concurrent delivery limits
  • Real-time performance: sub-second calculation for next-move decisions
  • Scalability: handle floor sizes up to 50×50 tiles (2,500 positions)
  • Support arbitrary floor layouts with blocked areas and multiple stations
  • Stateful operations: maintain session state for incremental "next step" calculations

Technical Challenges

  • Multi-agent pathfinding: Individual order optimization doesn't account for other paths - globally optimal solution is NP-hard
  • Deadlock prevention: Multiple orders competing for same empty tiles creates circular dependencies
  • Memory constraints: 50×50 floor = 2,500 tiles = 3.1M unique tile pairs for distance calculation
  • State management: Real-time "next step" mode requires maintaining warehouse state across API requests
  • Priority vs efficiency: Balancing high-priority orders with overall system throughput

Technical Solution

Key Design Decisions

1. IDA* over BFS/Dijkstra

Decision: Iterative Deepening A* with custom heuristics

Rationale: Better memory efficiency than BFS, optimal paths guaranteed, configurable iteration limits

Trade-off: Slightly slower than pure A* but uses ~90% less memory for large grids

2. Pre-calculated Distance Caching

Decision: Calculate and cache all tile-to-tile distances on floor registration

Rationale: O(1) distance lookups during pathfinding vs O(n) recalculation

Trade-off: Higher memory usage (~2MB for 50×50 floor) vs 200x faster path calculations

3. Two-Mode Operation: 'all' vs 'next'

Decision: Support both complete path calculation and single-step incremental mode

Rationale: 'all' mode for testing/simulation, 'next' mode for real-time production with visualization

Trade-off: More complex state management, but enables real-time oversight and debugging

4. Greedy Local Optimization vs Global

Decision: Greedy local optimization with conflict resolution instead of global multi-agent pathfinding

Rationale: 100x faster calculation (milliseconds vs seconds), deterministic behavior, testable/explainable

Trade-off: ~5% suboptimality vs theoretical global best, but no training data or RL infrastructure needed

IDA* Pathfinding with Custom Heuristics

Multi-component cost function balancing delivery distance, empty tile positioning, and future moves:

# Multi-component cost heuristic
def calculate_cost(order, station, empties):
    # Primary: Distance to target station
    cost_target = order.distance(station)

    # Secondary: Proximity to empty tiles (enables future moves)
    cost_empty = avg([order.distance(e) for e in empties]) / 5

    # Tertiary: Position empties near stations for return paths
    cost_empty_station = avg([station.distance(e) for e in empties]) / 10

    return cost_target + cost_empty + cost_empty_station

Deadlock Prevention System

Multi-layered conflict resolution ensures orders move safely without circular dependencies:

# Multi-layered conflict resolution
def allow_swap(self, swap, performed_swaps, other_orders):
    # Layer 1: Check conflict with already-performed swaps
    changed = set(swap).intersection(sum(performed_swaps, ()))
    if changed:
        return None  # Defer to next step

    # Layer 2: Don't move higher-priority orders
    swapping_higher_priority = any(
        order.priority < self.priority and order.rack_idx in swap
        for order in other_orders
    )
    if swapping_higher_priority:
        return None

    # Layer 3: Respect station-blocked empty tiles
    if set(swap).intersection(station_blocked_empties):
        return None

    return swap  # Safe to execute

Technical Challenges & Solutions

Challenge 1: Preventing Deadlocks in Parallel Movement

With 100+ orders moving simultaneously, naive approaches create deadlocks where orders block each other's paths. Root causes: multiple orders competing for same empty tiles, circular dependencies, station congestion.

Solution: Multi-layered conflict resolution system

  • Layer 1: Priority-based empty tile assignment - higher priority orders get first choice
  • Layer 2: Swap conflict detection - check if proposed swap affects already-committed swaps
  • Layer 3: Cross-order collision prevention - prevent lower-priority orders from moving higher-priority ones
  • Layer 4: Station-based blocking - track which empty tile is held by order occupying station

Outcome: Zero deadlocks in thousands of simulation runs

Challenge 2: Memory-Efficient Distance Calculation for Large Floors

50×50 floor = 2,500 tiles = 3.1M unique tile pairs. Recalculating distances on-demand: O(n) BFS per lookup. For typical order path (20 steps × 4 neighbors = 80 lookups): ~80 BFS calls resulted in 10+ second pathfinding times.

Solution: Pre-calculate all distances once on floor registration

  • BFS from each tile to all others: O(n²) one-time calculation
  • Bidirectional keys using frozenset({A, B}) - halves storage
  • Intermediate caching during BFS reduces redundant calculations
  • One-time cost: 250s for 50×50 floor, cached for all subsequent operations

Outcome: 200x speedup (from 10s to 50ms per pathfinding operation)

Challenge 3: Balancing Speed vs. Optimality for Multiple Orders

Finding globally optimal solution for N simultaneous orders is NP-hard. Individual IDA* pathfinding doesn't account for other orders' paths, creating suboptimal traffic patterns.

Solution: Hybrid strategy combining local optimization with global heuristics

  • Global empty distribution: assign empty tiles considering distance from order to empty and empty to destination
  • Individual path optimization: each order calculates optimal path given its assigned empty using IDA*
  • Execution-time conflict resolution: orders propose swaps in priority order, conflicts deferred to next step

Trade-off Analysis: Greedy local optimization vs global (RL/multi-agent pathfinding). Chose local: 100x faster (milliseconds vs seconds), deterministic, ~5% suboptimality vs theoretical best acceptable for practical use

Outcome: Sub-second response times with near-optimal paths

Challenge 4: State Management Across "Next Step" API Calls

Real-time mode requires maintaining warehouse state across multiple API requests while supporting concurrent testing sessions. Each "next step" call must resume from exact previous state (order positions, empty tiles, station occupancy, priorities).

Solution: Stateful controller with unique session IDs

  • Class-level dictionary: {controller_id: Controller_instance} for session management
  • UUID-based session IDs for unique identification
  • History tracking: empties_history, orders_history, station_history arrays for state reconstruction
  • Cleanup endpoints for session management and memory control

Outcome: Enables PyGame visualization to step through algorithm execution frame-by-frame for debugging and demonstration
